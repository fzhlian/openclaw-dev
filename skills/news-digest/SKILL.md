# news-digest

按以下流程执行。

## 需求采集（先于检索）

先把需求补齐成可执行参数，不完整时不要直接开始抓取。

### 采集顺序

1. 追踪主题（关键词）
2. 来源范围（站点/媒体/频道）
3. 时间范围（如 24h / 7d / 30d）
4. 更新频率（一次性 / 每日 / 每周）
5. 返回规模（条数上限）
6. 输出偏好（语言、结构、是否按主题分组）

### 追问模板（按缺项补问）

- 主题不清：
  - `你最想追踪哪几个主题？可先给 3-8 个关键词。`
- 来源不清：
  - `你希望限定哪些站点/媒体？如果不限定，我可先给高相关来源建议。`
- 时间不清：
  - `你要看最近 24 小时、7 天，还是 30 天？`
- 频率不清：
  - `这是一次性检索，还是要做日报/周报模板？`
- 输出不清：
  - 默认使用 `摘要总览 + 逐条清单` 并显式告知；只有用户明确要求其他结构时，再切到对应模式

### 采集完成确认

在首次检索前，用 1 段“参数确认”回显。
可调用 `scripts/intake_check.py` 根据当前用户输入自动生成“缺项追问清单 + 参数确认块”。

- 关键词：...
- 网站：...
- 时间范围：...
- 频率：...
- 结果数：...
- 输出模式：...
- 输出语言：...

执行约束：

1. 用户输入明显不完整时，先运行 `scripts/intake_check.py` 或按同等逻辑补问。
2. 若输出里仍存在“（待确认）”字段，先发追问或参数确认，不要直接开始检索。
3. 只有在用户明确确认参数后，才进入查询生成与抓取阶段。
4. 若时间范围或输出模式使用了默认值，要在确认块中显式告知。
5. 若用户明确说“执行一次”“跑一遍”“先发一版”，应将频率归一化为 `一次性`，并在确认块中显式回显，不要省略该字段。
6. 若用户把频率写成 `每 周`、`每 日`、`一 次 性` 这类带空格的自然写法，也应先归一化为标准频率名。
7. 当前最终摘要模板仅支持中文输出；若用户指定其他输出语言，应明确说明当前 skill 仅支持中文，不要假装已切换语言。
8. 若用户用“总览+逐条”“总览＋逐条”“按主题分组 + 逐条”等自然写法表达输出模式，应先归一化为标准模式名，再进入确认与渲染。
9. 若用户用 `24h`、`7d`、`30d`、`1d`、`14d`，或直接说 `7 天`、`30 天`、`24 小时`、`最近7天`、`最近24小时`、`最近1周`、`过去1周`、`近一周`、`1周`、`最近1个月`、`近1个月`、`最近一月`、`近一月`、`今天`、`昨天`、`本周`、`本星期`、`本礼拜`、`本月`、`这周`、`这星期`、`这礼拜`、`这一周`、`这个月`、`这月`、`这一月`、`上周`、`上星期`、`上礼拜`、`上一周`、`上一星期`、`上个月`、`上月`、`上一月`、`上一个月` 等简写，应先归一化为 `最近 24 小时`、`最近 7 天`、`最近 30 天`、`最近 1 天` 等标准写法，再进入确认与渲染。
10. 最终摘要里的“检索参数”应回显归一化后的参数值：关键词用 `、` 连接，网站统一显示为主域名，不要原样回显逗号串或完整 URL。
11. 关键词、网站等列表参数若使用全角逗号 `，`、中文顿号 `、`、中文分号 `；`、英文分号 `;`、竖线 `|`、全角斜杠 `／`，或带空格斜杠 ` / `，也应按普通逗号处理，不要把整串误当成单个词或单个站点。
12. 关键词若夹带句末标点，如 `OpenAI。`、`Gemini！`，也应先去掉尾部标点，再进入确认、查询生成和最终参数区。
13. 时间范围、频率、输出模式若仅多了句末标点，如 `最近7天。`、`执行一次。`、`总览+逐条。`，也应先去掉尾部标点，再进入确认和最终渲染。
14. 输出语言若仅多了句末标点，如 `中文。`，也应先去掉尾部标点，再按当前仅支持中文的规则处理。
15. 时间范围、频率、输出模式、输出语言若被引号或括号包裹，如 `“最近7天”`、`（执行一次）`、`《总览+逐条》`、`“中文”`，也应先去掉外层包裹符号，再进入确认和最终渲染。
16. 排除词若夹带句末标点，如 `ads。`、`tracking！`，也应先去掉尾部标点，再拼进查询，不要把标点原样带进 `-"词"` 片段。
17. 关键词、站点、排除词若被引号或括号包裹，如 `“OpenAI”`、`（openai.com）`，也应先去掉外层包裹符号，再进入确认、查询生成、过滤与最终参数区。

## 输入参数规范

先明确输入参数，缺失时先补问。

- 关键词：必填，支持 1 个或多个；建议优先使用名词短语
- 网站范围：必填，支持单站或多站；统一写域名，如 `openai.com`
- 若用户提供的是完整 URL 或带 `www.` 的地址，先归一化为主域名；若提供的是媒体名而非域名，先归一化为域名后再执行；对 `BBC`、`RFI`、`纽约时报`、`DW`、`华尔街见闻` 这类常见媒体名可直接归一化，无法确定时先补问，不要把未归一化的裸名称直接传给底层脚本
- 时间范围：可选，默认最近 7 天
- 返回条数：可选，默认 5，最大 20
- 更新频率：当前仅支持 `一次性`、`每日`、`每周`
- 输出语言：当前仅支持中文，默认中文
- 输出模式：当前仅支持“摘要总览 + 逐条清单”或“按主题分组+逐条”，默认前者

当用户输入不完整时，按以下顺序补问：

1. 缺关键词时，先问追踪主题
2. 缺网站时，先问来源范围
3. 缺时间范围时，使用默认值并显式告知

## 快速执行清单

1. 归一化参数：关键词、站点、时间范围、条数
2. 仅在必要时扩词：当原始关键词结果过少，或用户明确要求扩大召回时，再增加同义/近义/常见表述
3. 仅在必要时翻译检索词：当目标站点语言与输入语言不一致时，再补充对应语言关键词
4. 生成查询：优先 `site:domain keyword`（可用脚本）
5. 串行检索：按“站点 × 关键词”逐条检索，不使用超大 OR 混搜
6. 严格域名过滤：仅保留 URL 主域名命中目标站点的结果
7. 过滤去重：去广告页、导航页、弱相关，按 URL/标题去重
8. 摘要输出：先总览，再逐条
9. 统一中文模板：参数区、结构标题与说明统一为中文；摘要正文优先使用 `snippetZh` / `summaryZh`，若上游未提供中文摘要，则退回原始摘要文本
10. 标注局限：来源受限、时间缺失、覆盖不足

## 执行流程

1. 确认参数
   - 将关键词拆分为可检索短语
   - 将网站输入统一转成域名列表
   - 缺项时优先调用 `scripts/intake_check.py` 生成追问与参数确认块

2. 生成查询
   - 参数确认后，再调用 `scripts/build_query.py` 生成查询列表
   - 优先组合检索词：`site:<域名> <关键词>`
   - 默认只用原始关键词；仅在召回不足或用户明确要求扩大召回时，才显式开启 `--expand`
   - 仅在目标站点明显以英文为主时，才显式开启 `--auto-english`
   - 关键词多项时，按主题分批检索再合并去重
   - 若脚本输出了按站点拆分的 `keywordPlan`，后续检索应按该计划执行，不要把所有词无差别混成一个超大 OR 查询

3. 检索候选内容
   - 按“站点 × 关键词”串行检索，不使用超大 OR 混搜
   - 先跑原始词；不足时再重跑扩词版本
   - 若英文站点召回不足，再补跑英文化关键词版本

4. 提取与筛选
   - 先将外部检索结果归一化为统一结果对象；字段要求见 `references/result-schema.md`
   - 最少整理出：`title`、`url`、`snippet`；能补齐时再补 `publishedAt`、`sourceDomain`
   - 若 `sourceDomain` 缺失，应从 URL 提取；若 `publishedAt` 缺失，应保留为空并在最终输出中标注“时间未标注”
   - 可先调用 `scripts/filter_results.py` 对候选结果做域名过滤与基础去重；当输入仍是 `description` / `summary` / `published_at` / `domain` 等别名字段时，显式加 `--normalize`
   - 域名校验：仅保留目标域名（含其同主域子域名）结果
   - 去重：按 URL 和标题近似度去重
   - 过滤：广告页、导航页、弱相关内容
   - 摘要阶段只基于过滤后的结果，不要直接对原始搜索结果混合总结

5. 输出结果
   - 先给“摘要总览”（3-6 条）
   - 再给“逐条摘要清单”
   - 每条包含标题、1-3 句摘要、来源站点、发布时间、链接
   - 可调用 `scripts/render_digest.py` 将过滤后的标准结果直接整理为最终 Markdown 结构；当用户要求“按主题分组+逐条”时，结果中每条都应带有 `topic` / `queryTopic` / `keyword` / `query` 之一，再显式传 `--output-mode "按主题分组+逐条"`
   - 参数区、结构标题与说明统一输出为中文；摘要正文优先使用 `snippetZh` / `summaryZh`，若上游未提供中文摘要，则退回原始摘要文本
   - 最终渲染时，来源优先显示 `matchedDomain`，其次 `sourceDomain`；若两者都缺失但 `url` 存在，应从 URL 提取主域名，不要落成“来源未标注”
   - 以上来源回填规则对“摘要总览”和“文章清单”都应一致，不要出现总览无来源、逐条才有来源的漂移
   - 若某条结果拿不到可直接落地的原始 URL，不要把它继续保留在“文章清单”里；先丢弃该条，必要时改走降级输出

## 失败降级策略

- 单个站点不可访问：标注“来源受限”，继续处理其他站点
- 某关键词无结果：保留该关键词，并标注“未检索到高相关结果”
- 发布时间缺失：标注“时间未标注”，不自行推断
- 检索结果过少：按以下顺序回退，不要一次性全部放宽：
  1. 保持站点不变，先放宽时间范围
  2. 仍不足时，对原有关键词显式开启扩词
  3. 仅对英文主站点显式开启英文化关键词
  4. 仍不足时，再减少站点限制或建议用户补充来源
- 仍无法形成有效摘要：只输出“检索参数 + 已发现结果 + 局限与建议”，不编造结论
- 若检索结果只能确认“主题方向”或只有搜索摘要、却拿不到稳定原始链接，不输出正常“文章清单”；改为“检索参数 + 已发现结果 + 局限与建议”

## 引用与可追溯性

- 逐条摘要必须附原始链接，不输出“无来源结论”
- 若多篇来源支撑同一结论，至少给 2 条代表链接
- 对转载或二手解读，优先回溯到原始发布页
- `链接` 字段必须是可直接打开的原始 URL；不要用“暂无”“未稳定返回链接”“仅确认主题方向”等占位文本伪装成逐条条目

## 质量要求

- 不编造未检索到的信息
- 对结论和来源保持一一对应
- 对时间、来源不明的信息明确标注
